1. System Architecture Overview

The Central Intelligence system is a distributed AI automation platform built on two complementary components:

Core Architecture

Backend: Python 3.11+ | FastAPI | Claude SDK
Frontend: Next.js 14+ | TypeScript | Vercel
Database: Supabase PostgreSQL | 26 Tables | RLS Policies
Queue: Celery + Redis for async task execution

Key Design Principles

  • Database-Agnostic: Repository Pattern ensures all data access flows through a standardized interface. Swappable storage backends (PostgreSQL, MySQL, MongoDB) without agent code changes.
  • Async-First: All I/O operations (database, API calls, message queues) use async/await for non-blocking scalability.
  • Role-Based Access Control: Supabase RLS policies enforce row-level authorization. Users see only data they own or are explicitly granted.
  • Event-Driven: WebSocket subscriptions enable real-time UI updates. FastAPI webhooks trigger agent workflows asynchronously via Celery.

2. Agent Hierarchy & Specialization

Central Intelligence operates as a multi-tier agent system, mirroring organizational structure:

Tier 1: Central Intelligence (Orchestrator)

Role: Intent interpretation, workflow routing, chat inference via Claude SDK. Receives user input (dashboard chat, API requests) and routes to appropriate Director or Specialist.

Tier 2: 3 Directors

  • Marketing Director (CI-MKT-01): Content calendars, social metrics, audience targeting, brand voice
  • Sales Director (CI-SAL-01): Lead scoring, pipeline management, objection handling, conversion optimization
  • Fulfillment Director (CI-FUL-01): Member onboarding, coaching logistics, accountability tracking, win capture

Tier 3: 15+ Specialists

Each Specialist owns a domain (social media, email, funnels, calls, etc.) and executes specialized tasks. Example: Marketing Social Specialist (CI-MKT-03) manages LinkedIn, Instagram, TikTok metrics and scheduling.

Tier 4: 7 Operators

Shared utility workers: transcription, ICP generation, offer creation, stats aggregation, error recovery, data sync, webhook relay.

3. Data Schema Overview

Central Intelligence consolidates three data sources into a unified PostgreSQL schema: 26 tables organized into operational, intelligence, configuration, and audit tiers.

Design Philosophy

The schema enforces the Repository Pattern: all database operations flow through type-safe Repositories. Agents never execute raw SQL. This ensures:

  • Swappable Storage: Implement a new Repository concrete class → instant MySQL/MongoDB support
  • Consistent Contracts: All CRUD operations return predictable Pydantic schemas
  • Centralized Access Control: Only Repository classes hold credentials
  • Async-First: All methods use async/await for non-blocking I/O

Core Strategy: Soft Deletes & Multi-Tenant Isolation

  • Soft Deletes: Records marked deleted_at instead of hard-deleted. Enables data recovery and audit trails.
  • RLS Policies: Supabase enforces row-level security so users see only their data.
  • Idempotency Keys: Deduplication table prevents duplicate inserts on retried requests (24-hour window).

4. Repository Pattern & Data Access

All database operations flow through typed Repository classes. Agents depend on Repository interfaces, never raw SQL.

Base Repository Interface

class RepositoryBase(Generic[T]):
    async def get(id: Any) -> Optional[T]
    async def list(skip: int, limit: int, filters: Dict) -> List[T]
    async def create(obj_in: CreateSchema) -> T
    async def update(id: Any, obj_in: UpdateSchema) -> Optional[T]
    async def delete(id: Any) -> bool
    async def soft_delete(id: Any) -> Optional[T]  # Marks deleted_at
    async def exists(id: Any) -> bool

Concrete Implementation Example

class LeadRepository(RepositoryBase[Lead]):
    async def get_by_email(email: str) -> Optional[Lead]:
        stmt = select(Lead).where(
            and_(Lead.email == email.lower(), Lead.deleted_at == None)
        )
        result = await self.db_session.execute(stmt)
        return result.scalars().first()

    async def list_by_status(status: str, skip: int, limit: int):
        stmt = select(Lead).where(
            and_(Lead.status == status, Lead.deleted_at == None)
        ).offset(skip).limit(limit)
        result = await self.db_session.execute(stmt)
        return result.scalars().all()

Key Guarantee: Similar repositories exist for Member, Call, Insight, ContentIdea, and all 26 tables. Agents call repository methods; never databases directly.

6. Authentication & Authorization

Two-Layer Auth Model

Layer 1: User → App (Supabase Auth)

Users authenticate via email/password or OAuth provider. Supabase Auth issues a JWT stored in an HttpOnly cookie.

Layer 2: App → API (Bearer Token)

Every FastAPI request includes the JWT in the Authorization header:

Authorization: Bearer <supabase-jwt-token>
X-User-Id: user_abc123
X-Request-Id: req_xyz789
Content-Type: application/json

Required Headers

Header Required Description
Authorization Yes Bearer token from Supabase Auth
Content-Type Yes application/json
X-User-Id Yes* User ID from JWT claims (*except /auth and /health)
X-Request-Id Recommended UUID for request tracing and idempotency
X-Idempotency-Key Optional UUID to prevent duplicate processing (5-min window)

Security & Authorization Layers

  • Supabase JWT: Users authenticate with email/password. Supabase issues signed JWT. Token claims include user ID, email, role.
  • FastAPI Middleware: Every request verifies JWT signature and extracts user context. Failed verification returns 401 Unauthorized.
  • Row-Level Security (RLS): Supabase PostgreSQL enforces RLS policies on all queries. Users see only rows matching their user_id.
  • Role-Based Access: Roles (admin, coach, agent, viewer) control feature access. Admin-only endpoints require role = 'admin'.

Credential Rotation Strategy

  • API Keys: Stored as hashed values in database. Never logged or exposed in error messages.
  • Database Credentials: Managed by Supabase. Automatic rotation available via dashboard.
  • JWT Expiration: Supabase JWTs expire in 1 hour. Client refreshes token using refresh token endpoint.

HTTP Status Codes & Error Codes

Status Code Meaning
400 VALIDATION_ERROR Request body failed Pydantic validation
401 UNAUTHORIZED Missing or invalid Bearer token
403 FORBIDDEN Token valid but user lacks permission
404 NOT_FOUND Resource does not exist
409 CONFLICT Duplicate resource or version conflict
429 RATE_LIMIT_EXCEEDED Too many requests in time window
500 INTERNAL_ERROR Unexpected server-side failure
503 SERVICE_UNAVAILABLE API temporarily unavailable

Structured Logging

All errors logged via Sentry with structured context: user_id, request_id, endpoint, error_code, stack_trace. Enable Sentry integration for real-time alerting on critical errors.

Error Recovery Agent

Operator CI-CORE-05 (Error Handler) monitors error_log table and automatically retries failed tasks with exponential backoff. Critical failures escalate to admin queue.

9. Security & Operations

Authentication & Authorization Layers

  • Supabase JWT: Users authenticate with email/password. Supabase issues signed JWT. Token claims include user ID, email, role.
  • FastAPI Middleware: Every request verifies JWT signature and extracts user context. Failed verification returns 401 Unauthorized.
  • Row-Level Security (RLS): Supabase PostgreSQL enforces RLS policies on all queries. Users see only rows matching their user_id.
  • Role-Based Access: Roles (admin, coach, agent, viewer) control feature access. Admin-only endpoints require role = 'admin'.

Request Signing & HMAC (Optional)

For high-security workflows, clients can sign requests with HMAC-SHA256 using a shared secret. FastAPI middleware verifies signature before processing.

Data Protection

  • All data in transit over HTTPS with TLS 1.3
  • At-rest encryption via Supabase default configuration
  • Passwords hashed with bcrypt via Supabase Auth
  • API keys and credentials stored in .env (never committed)

Observability & Monitoring

  • Sentry captures exceptions, performance issues, and error trends
  • Custom agent telemetry tracks Claude SDK usage (tokens, latency)
  • Structured logging to error_log table for debugging
  • Health check endpoint (GET /health) monitors API availability
