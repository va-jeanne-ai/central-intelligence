Per-Rep Scope & Channel-Mix Analytics

Proposal — not yet built 2026-07-08 Extends the statistical recommendation engine (north star §5A/§7)

Turn the engine's question from “how is the company doing?” into “which rep and which channel needs attention?” — using only data that is already in the database, and reusing every layer of the existing pipeline.

1Why this, why now

The engine currently writes every metric snapshot with scope='global', even though the metric_snapshots schema (and its unique constraint) was explicitly designed for finer scopes. Meanwhile the single richest dataset in the app — 49,649 sales activities attributed to 7 named reps — contributes nothing to the engine. Every number below was queried read-only from the live app DB on 2026-07-08.

4Backend changes

4.1 Registry — a second SQL contract for rep fan-out

Add one optional field to the Metric dataclass. Where the global contract returns one (value, sample_size) row, rep_sql returns one row per rep in a single GROUP BY query — no N+1. 

@dataclass(frozen=True)
class Metric:
    ...
    # Optional per-rep fan-out. Contract: SELECT rep_id, value, sample_size
    # GROUP BY rep_id, honoring the same (:since IS NULL OR ...) window rule.
    rep_sql: TextClause | None = None

Metrics that gain rep_sql (all four have a verified rep_id column):

  • sales.avg_call_score — AVG(score) ... GROUP BY rep_id on sales_call_scores
  • sales.closed_sales_count / sales.revenue_collected — on closed_sales
  • fulfillment.open_coaching_strikes — on sales_coaching_strikes
  • sales.outbound_volume — new, see below

Two new global channel metrics (same one-row contract as today):

Metric(key="sales.outbound_volume", area="sales", unit="count", higher_is_better=True,
       label="Outbound Activity Volume",
       sql=...,      -- COUNT(*) of *_outbound activities in window; rep_sql groups it by rep_id
)
Metric(key="sales.channel_response_rate", area="sales", unit="ratio", higher_is_better=True,
       label="Outbound Response Rate",
       sql=...,      -- inbound / NULLIF(outbound, 0) over the window (global only —
)                    -- inbound rows carry no rep_id, so this metric has NO rep_sql)

4.2 Snapshots — fan-out write

After the existing global upsert, run rep_sql once per metric-window and upsert each row with scope='rep:<rep_id>'. The unique constraint (metric_key, "window", scope, captured_date) already accommodates this — no migration needed on metric_snapshots. 

if metric.rep_sql is not None:
    for row in db.execute(metric.rep_sql, {"since": since}).mappings():
        db.execute(_UPSERT_SCOPED, {..., "scope": f"rep:{row['rep_id']}",
                                    "value": row["value"], "sample_size": row["sample_size"]})

Volume math: 4 windows × ~7 reps × ~5 rep-scoped metrics ≈ +140 rows/day on top of today's 48 — negligible. Reps with status='terminated' are skipped at capture time (history retained, no new rows).

4.3 Trends — already scope-ready

evaluate() is pure arithmetic over a series; the series query just gains WHERE scope = :scope (default 'global', today hardcoded). The MIN_SAMPLE = 30 significance gate does exactly the right thing per rep: Ioana's 4 scored calls read insufficient_data, never a fake trend — the engine's no-fabrication contract extends to reps for free.

4.4 Recommendations — one migration, rep-aware phrasing

  • Add scope column to recommendations (default 'global'); widen the upsert conflict key from (metric_key, window) to (metric_key, window, scope). Single small alembic migration.
  • Phrase titles with the display name from sales_reps.full_name: “Nelson Figueria's Avg Call Score is declining …” — the numbers still come only from the evidence JSON, per the hypothesize-not-fabricate rule.
  • Guardrail against noise: rep-scoped findings cap at warn severity initially, and per-rep thresholds only fire when the rep's own sample_size ≥ MIN_SAMPLE.

4.5 API + CI chat

  • GET /analytics/team — the leaderboard: one row per active rep × rep-scoped metrics, with latest value, verdict, and sample size (assembled from snapshots + evaluate(), same as /analytics/trends).
  • scope query param on the existing /analytics/trends and /analytics/recommendations.
  • get_analytics_verdicts CI-chat tool gains an optional rep argument (name or rep_id — resolved against sales_reps, including historical_aliases). Stays CI-only; directors remain department-scoped per the architecture rule.

6Rollout — three branches, in order

PR 1 · backend
feat/rep-scope-snapshots

Registry rep_sql + 2 channel metrics; snapshot fan-out; recommendations scope migration; rep-aware phrasing; tests (registry invariants extend automatically; add fan-out + lifecycle-per-scope tests). Ships silently — no UI change, snapshots start accumulating immediately.

PR 2 · api + chat
feat/team-analytics-api

GET /analytics/team, scope params, CI-chat rep argument with alias resolution. Verify with authenticated curl against real snapshots from PR 1.

PR 3 · frontend
feat/insights-team-tab

Team tab (leaderboard + KPI row + channel mix card + rep detail drawer), built on the atomic library. By the time this merges, PR 1 has been accumulating per-rep history for days — verdicts render real on day one.

Estimated effort: PR 1 ≈ half a day, PR 2 ≈ 2–3 h, PR 3 ≈ a day. Ship PR 1 early — history only accumulates from the day it deploys.

7Risks & open questions

  • Recommendation noise: 7 reps × several metrics can multiply findings. Mitigated by the MIN_SAMPLE gate, the warn severity cap, and the existing auto-resolve lifecycle. Tune after a week of real output.
  • Terminated reps: skipped at capture; history retained. Confirm this matches how Greg wants departures handled.
  • Visibility: should every staff login see the full leaderboard (scores + strikes), or managers only? The app has no role-based UI gating today — if this needs restricting, that's a separate (small) feature. Needs Greg's call.
  • Inbound attribution: per-rep response rate stays impossible until the upstream WGR pipeline attributes inbound rows to reps. Worth flagging to the client — it's their data shape, not ours.
  • Probation cohort: sales_reps carries probation windows (Jun 1 → Aug 10 for 3 reps) — a natural future extension is a probation scorecard against targets, but that needs targets defined first. Out of scope here.
