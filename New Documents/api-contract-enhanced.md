## Overview

The Central Intelligence system is a business automation platform organized as a "Central Intelligence" of specialized AI workers. The Next.js web application communicates exclusively with FastAPI REST endpoints backed by Supabase. The web app never connects to databases directly — all data access and business logic is handled by FastAPI Central Intelligence agents and the repository pattern.

```
Next.js Web App
     |
     | HTTPS + Bearer Token (Supabase JWT)
     v
FastAPI REST Endpoints (api.centralintelligence.dev/v1)
     |
     +---> Central Intelligence (orchestration + AI chat via Claude SDK)
     +---> Directors (Marketing, Sales, Fulfillment department-level AI)
     +---> Marketing Specialists (social, email, funnels, ads, DM, offers)
     +---> Sales Specialists (leads, appointments, sales calls)
     +---> Fulfillment Specialists (members, coaching, accountability, tech SOS)
     +---> Operators (transcriber, ICP generator, offer generator, stats updaters)
     +---> Cross-domain feeds (wins, pain points, content ideas, objections, goals)
     +---> Admin & Dashboard
     +---> Central Intelligence (CI endpoints)
     |
     v
Supabase (PostgreSQL) + Repository Pattern
```

### Central Intelligence Subsystem

```
Central Intelligence (Python AI Agents + Supabase)
     |
     | Claude SDK Skills (direct Supabase access via Repository)
     v
CI Supabase Database (9 tables: calls, insights, content_ideas, market_signals...)
     |
     | FastAPI webhook endpoints (maintains database-agnostic pattern)
     v
FastAPI REST Endpoints → Next.js Web App
```

Central Intelligence is a marketing intelligence subsystem with its own Supabase database. The web app accesses CI data through FastAPI REST endpoints, maintaining the database-agnostic architecture through the repository pattern.

**Base URL**: `https://api.centralintelligence.dev/v1`

All request and response bodies use `Content-Type: application/json`.

## Authentication

Every request must include a Bearer token in the `Authorization` header. The token is a Supabase JWT (JSON Web Token) that authenticates the user and provides identity context.

```
Authorization: Bearer <supabase-jwt-token>
```

The JWT is issued by Supabase Auth and contains claims for user ID, email, and role. It is verified by the FastAPI middleware on every request using the Supabase client.

### Two-Layer Authentication Model

Authentication uses two layers:

1. **User → App** (Supabase Auth): Users authenticate to the web app via email/password or OAuth. The app receives a JWT from Supabase stored in an HttpOnly cookie. This layer is handled entirely by Next.js and Supabase Auth.

2. **App → API** (Bearer Token): The web app authenticates to FastAPI endpoints using the Supabase JWT. All requests include `X-User-Id` for audit purposes.

### Required Request Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Authorization` | Yes | `Bearer <supabase-jwt>` — JWT token from Supabase Auth |
| `Content-Type` | Yes | `application/json` |
| `X-User-Id` | Yes* | Authenticated user ID from Supabase token. *Except for `/auth/*` and `/health` endpoints. |
| `X-Request-Id` | Recommended | UUID for request tracing across Central Intelligences. Generated by the frontend. |
| `X-Idempotency-Key` | Optional | UUID to prevent duplicate POST/PUT processing. Honored for 5-minute window. |
| `If-Match` | Optional | `updatedAt` timestamp for optimistic locking on PUT requests. Returns 409 CONFLICT if stale. |

### Authentication Middleware (FastAPI)

```python
from fastapi import Depends, HTTPException
from supabase import Client

async def verify_jwt_token(authorization: str = Header(...)):
    """Verify Supabase JWT token and extract user claims."""
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Missing Bearer token")

    token = authorization[7:]

    try:
        # Supabase client verifies and decodes JWT
        payload = supabase.auth.get_user(token)
        return payload
    except Exception:
        raise HTTPException(
            status_code=401,
            detail="Invalid or expired token"
        )

# Use in endpoints
@app.get("/leads")
async def list_leads(user = Depends(verify_jwt_token)):
    # user contains: id, email, role, etc.
    pass
```

## Error Format

All error responses follow a consistent envelope. The `error.code` field is a stable machine-readable string that the web app can use for conditional handling. The `error.message` is human-readable and may change between releases.

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable description of what went wrong.",
    "field": "fieldName",
    "timestamp": "2026-03-04T12:00:00.000Z",
    "requestId": "req_abc123"
  }
}
```

### Standard Error Codes

| HTTP Status | Code | Meaning |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body or query params failed Pydantic validation |
| 400 | `MISSING_REQUIRED_FIELD` | A required field was absent |
| 400 | `INVALID_FIELD_VALUE` | A field value is outside the allowed set |
| 401 | `UNAUTHORIZED` | Missing or invalid Bearer token |
| 403 | `FORBIDDEN` | Token valid but user lacks permission for resource |
| 404 | `NOT_FOUND` | Resource does not exist |
| 409 | `CONFLICT` | Duplicate resource; request cannot be completed |
| 422 | `UNPROCESSABLE` | Request is syntactically valid but logically cannot be processed |
| 429 | `RATE_LIMIT_EXCEEDED` | Too many requests; see Rate Limiting section |
| 408 | `REQUEST_TIMEOUT` | Request exceeded the configured timeout |
| 409 | `VERSION_CONFLICT` | Record was modified since last read; re-fetch and retry |
| 413 | `PAYLOAD_TOO_LARGE` | Request body or uploaded file exceeds size limit |
| 500 | `INTERNAL_ERROR` | Unexpected server-side failure |
| 503 | `SERVICE_UNAVAILABLE` | API service temporarily unavailable |

### Error Response Example (FastAPI HTTPException)

```python
from fastapi import HTTPException

raise HTTPException(
    status_code=400,
    detail={
        "error": {
            "code": "VALIDATION_ERROR",
            "message": "Email must be a valid email address",
            "field": "email",
            "timestamp": datetime.utcnow().isoformat() + "Z",
            "requestId": request.headers.get("X-Request-Id", "unknown")
        }
    }
)
```

## Pagination Conventions

List endpoints that can return multiple records support cursor-based pagination using the following query parameters and response envelope.

### Query Parameters

| Parameter | Type | Default | Description |
|---|---|---|---|
| `page` | integer | `1` | 1-indexed page number |
| `limit` | integer | `20` | Records per page. Max `100`. |

### Response Envelope for Lists

```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 142,
    "totalPages": 8,
    "hasNextPage": true,
    "hasPreviousPage": false
  }
}
```

### Pydantic Model Example

```python
from pydantic import BaseModel
from typing import List, Any

class PaginationInfo(BaseModel):
    page: int
    limit: int
    total: int
    totalPages: int
    hasNextPage: bool
    hasPreviousPage: bool

class ListResponse(BaseModel):
    data: List[Any]
    pagination: PaginationInfo
```

## Rate Limiting

Requests are limited at the FastAPI middleware level using SlowAPI. The current configuration targets normal web app usage patterns.

| Scope | Limit | Window |
|---|---|---|
| Global (all endpoints) | 300 requests | 1 minute |
| Per endpoint | 60 requests | 1 minute |
| AI / analysis endpoints | 20 requests | 1 minute |

When the limit is exceeded, the response is `429 Too Many Requests` with appropriate headers and body.

### Rate Limiting Implementation (SlowAPI)

```python
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)
app.state.limiter = limiter

@app.get("/leads", dependencies=[Depends(limiter.limit("60/minute"))])
async def list_leads():
    pass

@app.post("/central-intelligence/chat", dependencies=[Depends(limiter.limit("20/minute"))])
async def central_intelligence_chat():
    pass
```

### Rate Limit Response Headers

```
X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1646079600
```

## 7. API Versioning Strategy

### Approach: URL-Based Versioning

The Central Intelligence API uses **URL path versioning** for clarity and backwards compatibility.

#### Version Format

```
https://api.centralintelligence.dev/v1/endpoint
https://api.centralintelligence.dev/v2/endpoint
https://api.centralintelligence.dev/v3/endpoint
```

Current active version: `v1` (2026-03-29 release)

#### How Versioning Works

- **New features** within a minor version (e.g., v1.0 → v1.1) are additive only and do NOT break existing clients
- **Breaking changes** trigger a new major version (e.g., v1 → v2)
- Old versions are supported for a minimum of **6 months** before deprecation
- Version sunset dates are published in release notes with at least 3 months notice

#### Major Version Changes (When to Increment)

Breaking changes include:

- Removing an endpoint
- Removing a required response field
- Changing a field's data type (e.g., `count: number` → `count: string`)
- Changing HTTP verb (GET → POST)
- Changing authentication mechanism
- Reordering array items in a way that breaks positional references

Minor changes (no version bump):

- Adding a new optional field to a response
- Adding a new endpoint
- Deprecating a field (still functional, but marked as deprecated in documentation)
- Adding a new optional query parameter

#### Example: Marketing Refactor (v1 → v2)

Suppose in Q3 2026 we want to restructure all marketing endpoints from `/marketing/*` to `/departments/marketing/*`.

**Before (v1)**:
```
GET /v1/marketing/summary
POST /v1/marketing/chat
```

**After (v2)**:
```
GET /v2/departments/marketing/summary
POST /v2/departments/marketing/chat
```

We announce:
- v1 will be supported until 2026-08-12
- v1 calls still work but log deprecation warnings in server logs
- v2 is the recommended version starting 2026-06-12
- Client migration guide published with exact mapping

---

### Deprecation Policy and Timeline

#### Three-Stage Deprecation

**Stage 1: Announcement** (3 months before removal)
- Deprecation notice added to API documentation
- Server logs include `X-Deprecation-Warning` header in responses
- Email notification sent to known API consumers
- Example: 2026-03-01 announcement of sunset date 2026-06-01

**Stage 2: Continued Support with Warnings** (3 months)
- Feature/endpoint still fully functional
- All requests include `X-Deprecation-Warning: true` header
- Deprecation message appears in API docs with clear link to migration guide
- Support tickets regarding deprecated features include migration recommendation

**Stage 3: Removal**
- Feature/endpoint no longer available
- Requests return `410 Gone` with explanation of replacement
- All users must have migrated to replacement

#### Deprecation Example: Old Auth Endpoint

Suppose we redesign auth. We might deprecate the old format:

```
2026-03-05: Deprecation announced. Old format still works.
            Header: X-Deprecation-Warning: endpoint /auth/old will be removed 2026-06-05

2026-03-05–2026-06-05: Old endpoint functional. All responses include deprecation header.
                       Migration guide published: "Updated Auth Flow (v1.1)"

2026-06-05: Endpoint removed. Requests return 410 Gone with link to migration guide.
```

## 8. WebSocket / Real-Time Events

### Overview

The Central Intelligence API supports WebSocket connections for real-time streaming of Central Intelligence chat responses and system events. WebSocket connections use the same JWT authentication as REST endpoints.

### WebSocket Connection

**Base URL**: `wss://api.centralintelligence.dev/ws/v1`

#### Initial Connection

```javascript
const ws = new WebSocket('wss://api.centralintelligence.dev/ws/v1');

// Authorization via header (or URL parameter)
ws.onopen = () => {
  ws.send(JSON.stringify({
    action: 'init',
    token: '<supabase-jwt>',
    userId: '<user-id>'
  }));
};

ws.onmessage = (event) => {
  const { action, data } = JSON.parse(event.data);

  if (action === 'ready') {
    console.log('Connected to Central Intelligence API:', data.serverId);
  }
};
```

#### Heartbeat (Keep-Alive)

```javascript
// Server → Client (every 30 seconds)
ws.onmessage = (event) => {
  const message = JSON.parse(event.data);

  if (message.action === 'ping') {
    ws.send(JSON.stringify({ action: 'pong' }));
  }
};
```

If server misses 2 consecutive pongs (60 seconds), connection is terminated.

#### Graceful Disconnect

```javascript
ws.send(JSON.stringify({ action: 'unsubscribe', channels: ['central-intelligence:chat'] }));
ws.send(JSON.stringify({ action: 'close' }));
// Server responds with close frame (code 1000)
```

### Reconnection Strategy

Client implements exponential backoff:

```javascript
let retryDelay = 1000; // 1 second
let maxRetries = 10;
let retryCount = 0;

function reconnect() {
  if (retryCount >= maxRetries) {
    console.error('Max reconnection attempts reached');
    return;
  }

  try {
    ws = new WebSocket('wss://api.centralintelligence.dev/ws/v1');
    retryCount = 0; // Reset on success
  } catch (err) {
    retryCount++;
    setTimeout(reconnect, retryDelay);
    retryDelay = Math.min(retryDelay * 2, 30000); // Cap at 30s
  }
}
```

### Error Handling Patterns in the Client

#### Typed Error Classes

```typescript
import {
  Central IntelligenceError,
  ValidationError,
  AuthenticationError,
  RateLimitError,
  NotFoundError,
  ConflictError
} from '@centralintelligence/client';

try {
  await client.leads.create({ name: 'Jane', email: 'jane@example.com', phone: '+1-555-0001', source: 'webinar' });
} catch (err) {
  if (err instanceof ValidationError) {
    // Handle validation errors (400)
    console.error(`Validation error on field: ${err.field}`);
  } else if (err instanceof AuthenticationError) {
    // Handle auth errors (401)
    console.error('Token expired or invalid. Please re-authenticate.');
  } else if (err instanceof RateLimitError) {
    // Handle rate limit (429)
    console.error(`Rate limit. Retry after ${err.retryAfter} seconds.`);
  } else if (err instanceof ConflictError) {
    // Handle conflicts (409)
    console.error('Record was modified. Please refresh and try again.');
  } else if (err instanceof Central IntelligenceError) {
    // Generic Central Intelligence error
    console.error(`Error [${err.code}]: ${err.message}`);
  } else {
    // Network or other errors
    console.error('Unexpected error:', err);
  }
}
```

The client retries requests with exponential backoff for transient errors:

```typescript
const client = new Central IntelligenceClient({
  baseUrl: 'https://api.centralintelligence.dev/v1',
  token: supabaseJwt,
  retryConfig: {
    maxRetries: 3,
```

**Retry behavior**:
- 1st attempt fails with 503 → wait 100ms → retry
- 2nd attempt fails with 503 → wait 200ms → retry
- 3rd attempt fails with 503 → wait 400ms → retry
- 4th attempt fails with 503 → throw error
